> 目标：把 PRD 落成可执行开发计划，明确阶段目标、模块依赖、里程碑、验收标准与 issue 拆分方式。

---

# 1. 开发总目标

在不先做“大而全”的前提下，分阶段验证这条产品线是否成立：

**玄学命格生成 → 抽卡 → 配装 → 自动战斗 → Boss → 榜单**

最终不是先做一个漂亮 demo，而是做一个：
- 能跑通核心闭环
- 能验证玩法是否上头
- 能验证命理是否真的进入系统
- 能验证榜单与刷图是否驱动留存

---

# 2. 开发原则

## 2.1 原则一：先闭环，后扩张
先做完整但小的 loop，不先铺太多系统。

## 2.2 原则二：先验证产品假设，后堆内容
优先验证：
- 玩家会不会因为命格差异改变 build
- 玩家会不会为了掉装和榜单继续刷
- 命理分榜是否有讨论价值

## 2.3 原则三：先做数据驱动骨架
尽量把这些做成可配置：
- 职业模板
- 掉率表
- Boss 配置
- 地图配置
- 榜单配置

## 2.4 原则四：MVP 不做重社交
先做排行榜和分享逻辑，不做聊天、公会、实时 PvP。

## 当前阶段（2026-03）

项目当前已经不再处于“先把 MVP 闭环搭起来”的阶段，而是进入：
**内容深化 + 数值调优 + 竞争/传播补强 + 上线前准备**。

当前验证/发射口径：
- 继续以现有 `Phase 2 + 本地 Node 后端` 作为主承载版本
- 目标不是再开新大系统，而是把现有主循环做得更厚、更稳、更适合小范围试玩/验证
- 为后续微信小游戏化与真实上线准备内容、节奏、榜单竞争感和基础商业化露出

# 5. 模块拆分

# 模块 A：命格生成模块

## 目标
把现有 BaZi 引擎转成游戏可用开局数据。

## 子任务
1. 生辰输入组件
2. BaZi 结果解析层
3. 命格 → 职业映射
4. 身强弱 → 承压模型映射
5. 喜用神 / 忌神 → build 适配标签
6. 开局数据结构定义

## 输出数据结构（建议）
- classId
- className
- dayMaster
- usefulGods
- tabooGod
- baseStats
- growthTags
- recommendedBuild

## 验收标准
- 同样输入始终输出一致
- 不同命格能明显形成不同开局
- 输出结果能直接被战斗/装备系统消费

---

# 模块 B：核心数值与战斗模块

## 目标
建立可刷图的战斗骨架。

## 子任务
1. 角色属性结构
2. 敌人属性结构
3. 五行克制计算
4. 自动战斗循环
5. 技能触发框架
6. Boss 机制框架
7. 战斗结算页

## MVP 范围
- 1 主角色 + 1 守护灵
- 3 类怪物（小怪/精英/Boss）
- 1 个主动技能槽
- 10~15 个神通原型里先接 5 个

## 验收标准
- 战斗结果可解释
- 不同 build 的输出/生存差异明显
- Boss 不是纯数值墙

---

# 6. 技术切片建议

## Slice 1：数据层先行
先定义 JSON / TS schema（即便不用 TS，也先把结构规范化）：
- class config
- equipment config
- skill config
- map config
- boss config
- leaderboard row

## Slice 2：模拟器先行
先做内部战斗模拟/数值测试工具，哪怕是网页调试页。
目的：
- 快速改参数
- 快速验证 build
- 不用每次都走完整 UI

## Slice 3：页面壳子后接系统
UI 不应先于系统。顺序应为：
数据结构 → 逻辑 → 页面壳子 → 动效。

# 9. 风险清单

## 风险 1：命理只停留在文案层
### 表现
职业名字换了，但玩法没变化。
### 对策
命理必须接入：
- 职业模板
- 装备适配
- Boss 克制
- 榜单分层

## 风险 2：抽卡和装备太像普通换皮
### 表现
只有名字玄学，实际体验无差异。
### 对策
加入：
- 命格适配
- 喜用神词条
- 忌神反噬
- 同命理榜

## 风险 3：战斗太无聊
### 表现
只是看数值跳动。
### 对策
Boss 必须有机制，神通必须有触发差异。

## 风险 4：范围失控
### 表现
什么都想做，MVP 做不完。
### 对策
死守 MVP：
- 3 图
- 2 池
- 1 主 Boss + 1 日 Boss
- 3 榜

# 10. 验收标准

## 第一阶段（闭环验收）
满足以下 6 条才算过关：
1. 能生成命格职业
2. 能抽卡并配装
3. 能刷图并掉装
4. 能强化并提升战力
5. 能打 Boss
6. 能进榜单并看见竞争

## 第二阶段（产品验收）
满足以下 5 条才算成立：
1. 新用户 5 分钟内进入爽点
2. 玩家理解命格差异
3. 玩家会为了掉装和抽卡继续刷
4. 玩家会因为榜单产生继续成长动机
5. 同日主榜能形成明显讨论感

# 11. 当前建议结论

## 当前主判断
当前项目判断不再是“要不要先做 Phase 1”，而是：
**MVP 主循环已经搭起来，接下来要把它打磨到接近 launch target 的可验证状态。**

## 接下来先做什么
按 launch target，优先进入以下 4 个工作包：

1. **内容扩充包**
   - 增加地图/Boss 层次
   - 增加命器/套装/签名追件
   - 增加更明确的 build 分路与 chase 目标

2. **数值与节奏调优包**
   - 持续校准掉落、强化、抽卡保底、每日奖励与 Boss 收益
   - 把 `抽卡 → 换装 → 强化 → 刷图 → Boss → 榜单` 的回流节奏继续拉顺

3. **竞争与传播包**
   - 强化榜单比较感、差距提示、回流建议与分享卡预备层
   - 补更明确的同命理竞争理由与“为什么值得晒/传”的产品化表达

4. **上线 / 商业化准备包**
   - 收紧首购、月卡、活动入口与露出策略
   - 梳理 launch 所需的配置化、数据口径、付费承接与内容排期准备

## 暂时仍不做什么
- 复杂社交系统
- 实时 PvP
- 公会
- 重型 meta 经济扩建
- 为了“看起来更多”而新开大系统

## 为什么
因为当前最重要的，不再是证明“闭环能不能搭起来”，而是验证：
**现有这套“玄学命格 + 抽卡刷装 + 命理分榜”主循环，是否足够有内容深度、竞争张力和上线前可调优性。**
